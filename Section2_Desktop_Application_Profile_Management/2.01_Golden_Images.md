## 1. Golden Imageとは

Golden Image（マスターイメージ）は、仮想デスクトップおよびRDSHホストのベースとなるテンプレートVMである。Horizon環境ではこのイメージからInstant CloneまたはFull Cloneが作成される。イメージの品質がログオン時間、ユーザー体験、ストレージ消費、運用コストのすべてに直結するため、OCEレベルのエンジニアにはイメージ設計の判断力が求められる。

### 1.1 Golden Imageの構成要素

| レイヤー             | 内容                                                    |
| -------------------- | ------------------------------------------------------- |
| Guest OS             | Windows 10/11 Enterprise、Windows Server 2019/2022/2025 |
| VMware Tools         | 最新バージョンのVMware Tools                            |
| Horizon Agent        | Horizon Agent + 必要なFeature Pack                      |
| App Volumes Agent    | App Volumes連携時に必要                                 |
| DEM FlexEngine       | DEM連携時に必要（Agent = FlexEngine）                   |
| 業務アプリケーション | ベースイメージに含めるアプリ（App Volumes非配信分）     |
| 最適化設定           | OSOT / 手動レジストリ最適化                             |

### 1.2 Instant Clone vs Full Clone — 選択判断フレームワーク

| 判断基準             | Instant Clone                            | Full Clone                 |
| -------------------- | ---------------------------------------- | -------------------------- |
| プロビジョニング速度 | 数秒（vmFork技術）                       | 数分〜数十分               |
| ディスク消費         | 差分ディスクのみ（小さい）               | フルサイズの独立ディスク   |
| 再起動時の動作       | ログオフ/再起動でリセット                | データ永続                 |
| ユーザーカスタマイズ | App Volumes Writable Volume / DEM で補完 | OS上に直接保存可           |
| パッチ適用           | Push Image → 全Clone再作成               | 個別VMにWSUSやSCCM適用可   |
| 永続ディスク         | Persistent Disk（オプション）            | 標準でデータ永続           |
| ライセンス           | KMS / AVMA 推奨                          | KMS / MAK 可               |
| 推奨ユースケース     | タスクワーカー、標準化VDI                | 開発者、永続データ保持必須 |

> **重要**: Instant Cloneが圧倒的に主流である。Full Cloneは「ユーザーがOSレベルのカスタマイズを必要とし、App Volumes / DEM では対応できない」場合に限って選択する。

## 2. Golden Image VMの作成手順

### 2.1 VM作成とOS準備

#### Step 1: vSphereでVMを作成

| パラメータ      | 推奨値                                                                   |
| --------------- | ------------------------------------------------------------------------ |
| Guest OS Type   | Windows 10 (64-bit) / Windows 11 (64-bit) / Windows Server 2022 (64-bit) |
| vCPU            | 2〜4（ワークロードに応じて）                                             |
| vMemory         | 4〜8 GB                                                                  |
| vDisk           | 60〜80 GB (Thin Provisioning)                                            |
| SCSI Controller | VMware Paravirtual (PVSCSI)                                              |
| Network Adapter | VMXNET3                                                                  |
| Video Memory    | 128 MB以上（マルチモニタ対応時は増量）                                   |

#### Step 2: OSクリーンインストール

- Windows 10/11 Enterprise LTSC/Enterprise を推奨
- Server OSの場合は **Desktop Experience** を有効化
- RDSH用の場合は **Remote Desktop Services** ロールをインストール

#### Step 3: VMware Toolsのインストール

- vSphere Clientから最新のVMware Toolsをマウントしてインストール
- カスタムインストールで不要コンポーネントを除外可能

#### Step 4: Windows Updateの適用

- 累積更新プログラムをすべて適用
- .NET Framework更新も適用
- 適用後に再起動、再度確認して追加分を適用

### 2.2 Horizon Agentのインストール

#### インストーラの実行

```
VMware-Horizon-Agent-x86_64-xxxx-x.x.x-yyyyyyyy.exe
```

#### インストール時のFeature選択

| Feature                        | 説明                       | VDI    | RDSH   |
| ------------------------------ | -------------------------- | ------ | ------ |
| Core                           | 必須コンポーネント         | ✅     | ✅     |
| USB Redirection                | USBデバイスリダイレクト    | 必要時 | 必要時 |
| Real-Time Audio-Video (RTAV)   | Webcam/マイクリダイレクト  | ✅     | ✅     |
| Client Drive Redirection (CDR) | クライアントドライブの共有 | 必要時 | 必要時 |
| Scanner Redirection            | スキャナリダイレクト       | 必要時 | 必要時 |
| Virtual Printing               | 仮想プリンタ               | ✅     | ✅     |
| Instant Clone Agent            | Instant Clone使用時に必要  | ✅     | ✅     |
| Horizon Performance Tracker    | パフォーマンス監視         | 任意   | 任意   |
| VMware Integrated Printing     | 統合印刷                   | ✅     | ✅     |

> **重要**: Instant Clone Agentは、Instant Clone Poolに使用するGolden Imageには**必ず**インストールする。未インストールの場合、プール作成時にエラーとなる。

#### サイレントインストール（自動化向け）

```powershell
VMware-Horizon-Agent-x86_64-xxxx-x.x.x-yyyyyyyy.exe /s /v"/qn VDM_VC_MANAGED_AGENT=1 ADDLOCAL=Core,USB,RTAV,ClientDriveRedirection,V4V,VmwVaudio,InstantClone"
```

主要なサイレントインストールプロパティ:

| プロパティ                 | 説明                                        |
| -------------------------- | ------------------------------------------- |
| `VDM_VC_MANAGED_AGENT=1`   | vCenter管理下のVM（Automated Pool用）       |
| `ADDLOCAL=`                | インストールするFeatureのカンマ区切りリスト |
| `SUPPRESS_RUNONCE_CHECK=1` | RunOnceレジストリチェックをスキップ         |

### 2.3 Omnissa Agentのインストール（順序が重要）

公式ガイドに基づくエージェントのインストール順序:

| 順序 | エージェント       | 説明                                                |
| ---- | ------------------ | --------------------------------------------------- |
| 1    | Horizon Agent      | 最初にインストール。Feature Packの選択を含む        |
| 2    | DEM FlexEngine     | Config Shareパスの指定が必要                         |
| 3    | App Volumes Agent  | App Volumes Manager URLを指定して登録               |

> **重要**: Agentのインストールは **Horizon Agent → DEM FlexEngine → App Volumes Agent** の順序で行う。OSOTのOptimize / Generalizeが完了した**後**、Finalizeの**前**にインストールする。

**Horizon Agent**: 最初にインストール。Instant Clone Agentなどの必要なFeature Packを選択。

**DEM FlexEngine**:

- DEM Management Consoleとは別コンポーネント
- Golden ImageにはFlexEngine（Agent）のみをインストール
- GPOまたはレジストリでConfig Share（`\\fileserver\demconfig`）パスを指定

**App Volumes Agent**:

- App Volumes Managerに登録するため、インストール時にManager URLを指定
- Golden Imageに含めるが、パッケージング用VMには別途インストール

## 3. OSの最適化

### 3.1 OSOT (OS Optimization Tool) の全体ワークフロー

OSOTはOmnissa公式の最適化ツールで、VDI / RDSH環境向けにWindows OSを最適化する。公式ドキュメントでは **Optimize → Generalize → Finalize** の3段階ワークフローが定義されている。

```
Golden Image VM (Audit Mode)
  ↓ (1) Optimize — 不要サービス/タスク/視覚効果の無効化
  ↓ (2) Generalize — デバイス暗号化/BitLocker無効化 + 汎用化処理
  ↓ (3) Microsoft Storeアプリのクリーンアップ
  ↓ (4) Omnissa Agentインストール（Horizon Agent → DEM FlexEngine → App Volumes Agent）
  ↓ (5) GPUドライバインストール（必要時）
  ↓ (6) Finalize — イベントログクリア、一時ファイル削除、最終封印
  ↓ (7) スナップショット作成 / VMテンプレート化
```

> **重要**: OSOTには **Optimize → Generalize → Finalize** の3ステップがある。GeneralizeはOptimizeの後・Agentインストールの前に実行する。Finalizeはすべてのインストール完了後の最終封印処理である。この順序を間違えると正常にCloneが作成されない。

#### Audit Mode（監査モード）

Golden Image作成時はWindowsの **Audit Mode** を使用する。Audit Modeでは、OOBE（初回セットアップ）を経由せずに管理者としてOSにログインし、カスタマイズ作業を行える。

- Windows セットアップ画面で `Ctrl + Shift + F3` でAudit Modeに入る
- Audit ModeではSysprep GUIが自動表示されるが閉じてよい
- 全カスタマイズ完了後にGeneralizeで汎用化する

#### OSOTの主要最適化カテゴリ

| カテゴリ         | 最適化内容                                               |
| ---------------- | -------------------------------------------------------- |
| Windows Services | 不要サービスの無効化（Windows Search, Superfetch, etc.） |
| Scheduled Tasks  | 不要スケジュールタスクの無効化                           |
| Visual Effects   | 視覚効果の抑制（アニメーション、透過効果）               |
| Windows Features | 不要なWindows機能の削除                                  |
| Registry         | VDI最適化レジストリの適用                                |
| Disk Cleanup     | 一時ファイル、キャッシュの削除                           |
| Store Apps       | 不要なUWPアプリの削除                                    |

#### Optimize実行手順

1. **OSOTの起動** — 管理者権限で `VMwareOSOptimizationTool.exe` を実行
2. **テンプレートの選択**:
   - `Omnissa Templates` > 該当OSバージョン用テンプレートを選択
   - VDI用とRDSH用で異なるテンプレートが用意されている
3. **Analyze** — 現在の設定と推奨設定の差分を分析
4. **Optimize** — 推奨設定を適用

#### OSOTテンプレートのカスタマイズ

- OSOTテンプレートはXMLファイルで提供される
- カスタムテンプレートを作成し、組織固有の最適化ポリシーを定義可能
- テンプレートをエクスポートして他環境に移植可能

### 3.2 Generalize（汎用化）

Generalizeは、Golden Imageを複数クローンで使用可能な汎用状態にする処理。OSOTの **Generalize** タブから実行する。

#### Generalize前の必須作業

| 作業                       | 説明                                                             |
| -------------------------- | ---------------------------------------------------------------- |
| デバイス暗号化の無効化     | Windows 11ではデフォルト有効。`Settings > Privacy & Security > Device encryption > Off` |
| BitLockerの無効化          | `manage-bde -off C:` → `manage-bde -status` で復号完了を確認    |

> **重要**: Windows 11環境でGeneralize前にデバイス暗号化/BitLockerを無効化しないと、Generalizeが失敗する。

### 3.3 Microsoft Storeアプリのクリーンアップ

Generalize後、Agentインストール前にMicrosoft Storeアプリを整理する。

```powershell
# プロビジョニング済みアプリの削除
Get-AppxProvisionedPackage -Online | Remove-AppxProvisionedPackage -Online

# インストール済みアプリの削除
Get-AppxPackage | Remove-AppxPackage
```

> **重要**: Microsoft Storeアプリの残存はプロビジョニング時のSysprep失敗の最も一般的な原因の一つ。

### 3.4 Finalize（最終封印）

Finalizeはすべてのインストール完了後に実行する最終処理。

| 処理               | 説明                                       |
| ------------------ | ------------------------------------------ |
| イベントログクリア | Windowsイベントログをすべてクリア          |
| 一時ファイル削除   | Temp, Prefetch, ダウンロードフォルダの削除 |
| ディスクデフラグ   | ストレージ効率化のためデフラグを実行       |
| Windows Update停止 | Finalize中のUpdate処理を停止               |

> **重要**: FinalizeはAgentインストールの**後**に実行する。Finalize後にソフトウェアをインストールすると一時ファイルが残留し、イメージサイズが肥大化する。

### 3.5 手動によるレジストリ最適化

OSOTでカバーされない項目や、より細かい制御が必要な場合に手動最適化を行う。

#### 重要なレジストリ最適化項目

**OneDrive同期の無効化**:

```
HKLM\SOFTWARE\Policies\Microsoft\Windows\OneDrive
"DisableFileSyncNGSC"=dword:00000001
```

**Cortana / Searchの無効化**:

```
HKLM\SOFTWARE\Policies\Microsoft\Windows\Windows Search
"AllowCortana"=dword:00000000
```

**Windows Consumer Featuresの無効化**:

```
HKLM\SOFTWARE\Policies\Microsoft\Windows\CloudContent
"DisableWindowsConsumerFeatures"=dword:00000001
```

**NtfsDisableLastAccessUpdate**:

```
HKLM\SYSTEM\CurrentControlSet\Control\FileSystem
"NtfsDisableLastAccessUpdate"=dword:80000003
```

**電源プランを高パフォーマンスに設定**:

```powershell
powercfg /setactive 8c5e7fda-e8bf-4a96-9a85-a6e23a8c635c
powercfg /hibernate off
```

### 3.6 Windowsサービスの最適化

以下のサービスは非永続VDI環境で無効化が推奨される:

| サービス名  | 表示名                                   | 無効化理由                   |
| ----------- | ---------------------------------------- | ---------------------------- |
| WSearch     | Windows Search                           | インデックス作成がIOPSを消費 |
| SysMain     | SysMain (Superfetch)                     | プリフェッチがメモリを消費   |
| DiagTrack   | Connected User Experiences and Telemetry | テレメトリ通信が不要         |
| MapsBroker  | Downloaded Maps Manager                  | マップ不要                   |
| XblGameSave | Xbox Live Game Save                      | ゲーム関連不要               |
| OneSyncSvc  | Sync Host                                | 非永続環境では同期不要       |

### 3.7 ページファイルの最適化

| 環境                   | 推奨設定                                 |
| ---------------------- | ---------------------------------------- |
| Instant Clone (非永続) | 固定サイズ（例: 1024〜2048 MB）          |
| Full Clone (永続)      | システム管理またはRAMの1〜1.5倍          |
| RDSH                   | RAMの1倍（例: 32GB RAM → 32GB pagefile） |

> **重要**: Instant Clone環境ではページファイルが差分ディスクに書き込まれる。ページファイルサイズを適切に制限することで差分ディスクの肥大化を防ぎ、ストレージ消費を抑制できる。

## 4. Instant Clone用のGolden Image準備

### 4.1 Windows 11 Golden ImageとvTPM対応

Windows 11ではTPM 2.0が必須要件であるため、Golden Image作成時に特別な考慮が必要。

#### vTPMの取り扱いルール

| ルール | 説明 |
| ------ | ---- |
| Golden ImageにはvTPMを**付けない** | vTPMを付けて後で外すとデータロスと見なされ問題が発生する |
| vTPMはプールレベルで追加 | Desktop Pool設定の「Add virtual TPM」オプションで各Cloneに自動付与 |
| Windows 11インストール時のみ一時的に使用可 | セットアップ/24H2アップグレード時に必要な場合、完了後に削除 |
| OVFエクスポート前にvTPMを削除 | vTPMがあるとVM暗号化によりOVFエクスポートがブロックされる |

#### デプロイ方式別の対応

| デプロイ方式 | Golden Image | Clone側 |
| ------------ | ------------ | ------- |
| Instant Clone (非永続) | vTPMなしで作成 | Pool設定で「Add virtual TPM」を有効化 |
| Full Clone (永続) | vTPMなしで作成 | クローン後に個別VMにvTPMを追加 |

> **重要**: Windows 11 Golden Imageには**vTPMを含めない**のがベストプラクティス。Push Image時の「Add virtual TPM」オプションで各Instant Cloneに自動付与する。vTPMを一度付けてから外すとシステムに問題が発生する可能性がある。

### 4.2 ClonePrep vs Sysprep

| 項目               | ClonePrep                                    | Sysprep                             |
| ------------------ | -------------------------------------------- | ----------------------------------- |
| 用途               | Instant Clone専用の高速カスタマイズ          | 汎用的なWindowsカスタマイズ         |
| 速度               | 高速（SIDリセット + コンピュータ名変更のみ） | 低速（フルgeneralize + specialize） |
| ドメイン参加       | 自動（AD Computer Account再利用可）          | Customization Spec経由              |
| カスタムスクリプト | Pre/Post-synchronization scripts対応         | Customization Specで対応            |
| 推奨               | ✅ デフォルト推奨                            | 特殊要件時のみ                      |

> **重要**: ClonePrepはInstant Cloneの標準カスタマイズ方式で、Sysprepより大幅に高速。Sysprepを選択する理由は「組織のセキュリティポリシーでフルgeneralizeが必須」「特殊なカスタマイズスペックが必要」等に限られる。

### 4.3 Golden ImageのVMスナップショット

Instant Cloneプールを作成・更新する際に、Golden ImageのVMスナップショットを指定する。

**スナップショット作成のベストプラクティス**:

1. すべての更新・最適化が完了した状態でスナップショットを取得
2. VMの電源はON/OFF問わないが、電源OFF推奨（クリーンな状態）
3. スナップショット名に日付・バージョン・変更内容を含める

   - 例: `v2.1.0_2025-06-15_Win11-24H2-CumulativeUpdate`
4. スナップショットのチェーンを深くしすぎない（3〜5レベルまで）

### 4.4 Windows Activation

Instant Clone環境では以下のライセンス方式を使用:

| 方式                           | 説明                                                      |
| ------------------------------ | --------------------------------------------------------- |
| KMS (Key Management Service)   | KMSサーバーで一括アクティベーション。25台以上の閾値が必要 |
| AVMA (Automatic VM Activation) | Hyper-V専用。vSphere環境では使用不可                      |

- Golden ImageにはKMSクライアントキーを設定
- Instant Cloneは起動時に自動でKMSサーバーへアクティベーション要求
- MAKキーはInstant Cloneには使用不可（各Cloneが個別にアクティベーションを消費する）

## 5. Golden Imageのライフサイクル管理

### 5.1 更新フロー

```
Golden Image VM (電源ON)
  ↓ Windows Update / アプリ更新 / Agent更新
  ↓ OSOT Finalize (クリーンアップ)
  ↓ 電源OFF
  ↓ スナップショット作成
  ↓ Horizon Console > Pool > Push Image (新スナップショット指定)
  ↓ Instant Clone再作成（ログオフ時 or 即座）
```

### 5.2 Push Image操作

Instant Cloneプールの更新はHorizon Consoleの **Push Image** 機能で行う。

| パラメータ          | 説明                                     |
| ------------------- | ---------------------------------------- |
| Image               | 新しいGolden Image VM + Snapshot         |
| When to Push        | Immediately / At next logoff / Scheduled |
| Stop on first error | エラー発生時に残りのVMへの適用を停止     |
| Add virtual TPM     | vTPMの追加（VBS対応時）                  |

**Push Image Timing**:

| オプション     | 動作                                                |
| -------------- | --------------------------------------------------- |
| Immediately    | 接続中のユーザーにログオフを促し、即座にClone再作成 |
| At next logoff | ユーザーがログオフした時点で個別にClone再作成       |
| Scheduled      | 指定日時にPush Imageを開始                          |

> **重要**: Push Imageの「At next logoff」は段階的な更新に最適で、ユーザー影響を最小化できる。「Immediately」は緊急セキュリティパッチ適用時に使用する。

### 5.3 Day-2 Updates（運用中のイメージ更新）

Golden Imageの運用中更新（Day-2 Updates）は以下の手順で行う:

1. Golden Image VMの電源をON
2. Windows Updateの適用（累積更新プログラム）
3. Omnissa Agentの更新（Horizon Agent / App Volumes Agent / DEM FlexEngine）
4. Hypervisor Toolsの更新（VMware Tools等）
5. OSOT **Finalize** を再実行（一時ファイルのクリーンアップ）
6. 電源OFF → スナップショット作成
7. Push Imageで各プールに反映

> **重要**: Day-2 UpdatesではOSOTのOptimize/Generalizeを再実行する必要はない。Finalizeのみ再実行して一時ファイルをクリーンアップすればよい。

### 5.4 バージョン管理のベストプラクティス

**命名規則**:

```
<OS>-<用途>-v<major>.<minor>.<patch>
例: Win11-VDI-v2.1.0
    WinSrv2022-RDSH-v1.3.2
```

**変更管理テーブル例**:

| バージョン | 日付       | 変更内容            | スナップショット名 |
| ---------- | ---------- | ------------------- | ------------------ |
| v1.0.0     | 2025-01-15 | 初期イメージ作成    | v1.0.0_Initial     |
| v1.1.0     | 2025-02-01 | 累積更新KB5034441   | v1.1.0_CU-Feb2025  |
| v1.2.0     | 2025-03-01 | Horizon Agent 2412  | v1.2.0_Agent-2412  |
| v2.0.0     | 2025-06-01 | Windows 11 24H2更新 | v2.0.0_24H2        |

### 5.5 複数Golden Imageの管理戦略

| 戦略                       | 説明                                                       | 適用場面                     |
| -------------------------- | ---------------------------------------------------------- | ---------------------------- |
| 単一イメージ + App Volumes | ベースイメージを1つに統一し、アプリ差異はApp Volumesで対応 | 標準化が進んだ環境           |
| 部門別イメージ             | 部門ごとに異なるGolden Image                               | アプリ互換性の問題がある場合 |
| OS別イメージ               | Win10 / Win11 / Server OS 別                               | 移行期間中                   |

## 6. ベストプラクティスとアンチパターン

### ベストプラクティス

| #   | プラクティス                                                                 |
| --- | ---------------------------------------------------------------------------- |
| 1   | Golden Image専用VMを作成し、ユーザーデータや不要アプリを含めない             |
| 2   | OSOTで最適化後、手動で追加の最適化を検証適用                                 |
| 3   | App Volumesで配信するアプリはGolden Imageに含めない                          |
| 4   | スナップショット名に日付・バージョン・変更内容を含める                       |
| 5   | Push Imageの前にテストプール（小規模）で検証する                             |
| 6   | Horizon Agent / App Volumes Agent / DEM FlexEngineは常に最新バージョンを使用 |
| 7   | Golden Imageの変更履歴をドキュメント化する                                   |
| 8   | KMSライセンスを使用し、MAKキーは使用しない（Instant Clone環境）              |

### アンチパターン

| #   | アンチパターン                               | 問題点                               |
| --- | -------------------------------------------- | ------------------------------------ |
| 1   | 本番VMからGolden Imageを作成                 | 不要データ・設定が混入               |
| 2   | OSOTのすべての最適化を無検証で適用           | 必要なサービスが無効化される可能性   |
| 3   | スナップショットチェーンが10レベル以上       | パフォーマンス低下、ストレージ非効率 |
| 4   | アプリケーションをGolden Imageに詰め込みすぎ | 更新頻度増加、イメージ肥大化         |
| 5   | Instant CloneにMAKキーを使用                 | アクティベーション回数を消費し尽くす |
| 6   | テストなしで本番プールにPush Image           | 障害発生時に全ユーザーに影響         |

## 7. トラブルシューティング

| 症状                           | 原因候補                                          | 対処                                        |
| ------------------------------ | ------------------------------------------------- | ------------------------------------------- |
| ログオン時間が長い（30秒以上） | OSOT最適化未実施、不要サービス起動、GPOの過剰適用 | OSOT適用、サービス見直し、Login Monitor分析 |
| Push Image後にVMが起動しない   | スナップショット不整合、Agent未インストール       | スナップショット再作成、Agent確認           |
| アプリケーションが動作しない   | OSOT最適化で必要サービスを無効化                  | OSOTテンプレートを調整して再適用            |
| ストレージ使用量が急増         | ページファイル肥大、差分ディスク肥大              | ページファイル制限、Space Reclamation設定   |
| KMSアクティベーション失敗      | KMSサーバー到達不可、閾値未達                     | DNS SRV レコード確認、KMS閾値確認           |
